2.1 Pods — The Smallest Deployable Unit Essential
▼

📘 What is a Pod?

A Pod is the smallest and simplest unit in Kubernetes. It represents a single instance of a running process in your cluster.

Source: TechWorld with Nana (12:04)

Key Characteristics:
• Can contain one or more containers (usually one)
• Containers in a pod share network namespace (localhost)
• Containers in a pod share storage volumes
• Pods are ephemeral — they can be killed and recreated
• Each pod gets a unique IP address

🌍 Real-World Example

Think of a Pod as a shared apartment:

  • Containers = Roommates living together
  • They share the same address (IP)
  • They can talk to each other via localhost
  • They share common storage (volumes)

💻 Creating Pods

# Create a pod imperatively kubectl run nginx --image=nginx # Create from YAML kubectl apply -f - <

🛠️ Try It on AWS/EKS

Step 1: Create a simple pod

# Create pod kubectl run my-app --image=nginx # Watch pod creation kubectl get pods --watch # Check which node it's running on kubectl get pod my-app -o wide

💰 Cost: Negligible (uses existing EKS node resources)

⚠️ Common Pitfalls

1. Don't create pods directly in production
✅ Use Deployments, StatefulSets, or DaemonSets instead
❌ Pods created with kubectl run won't be recreated if they die

2. Multi-container pods are for tightly coupled apps only
Most apps should use 1 container per pod

3. Pods are ephemeral
Never store data directly in a pod — use volumes!

2.2 Pod Lifecycle & Status Important
▼

📘 Understanding Pod Lifecycle

Pods go through several phases during their lifecycle. Understanding these phases is critical for troubleshooting.

Pod Phases:
• Pending: Pod accepted but not yet running (downloading image, scheduling)
• Running: Pod bound to node, at least one container running
• Succeeded: All containers terminated successfully
• Failed: All containers terminated, at least one failed
• Unknown: Pod state cannot be determined

🔄 Pod Lifecycle Stages

1. Creation

When you create a pod:

  1. Pod object created in etcd (status: Pending)
  2. Scheduler assigns pod to a node
  3. Kubelet on the node pulls container image
  4. Container runtime starts the container
  5. Pod status changes to Running

2. Running

Pod is actively running. You can check logs, exec into containers, and monitor health.

# Check pod status kubectl get pods # Get detailed pod information kubectl describe pod my-pod # View pod logs kubectl logs my-pod # Execute command in pod kubectl exec -it my-pod -- /bin/bash

3. Termination

When a pod is deleted:

  1. Pod status marked as Terminating
  2. SIGTERM sent to containers (graceful shutdown)
  3. Grace period (default 30s) allows cleanup
  4. SIGKILL sent if containers still running after grace period
  5. Pod removed from API server

⚠️ Common Pitfalls

1. Pending pods don't always mean problems
Pending is normal during image download or node selection

2. Failed pods don't auto-restart by default
Use restartPolicy to control restart behavior

3. Terminating pods can get stuck
Use kubectl delete pod --force --grace-period=0 as last resort
2.3 ReplicaSets — Ensuring Replicas Important
▼

📘 What is a ReplicaSet?

A ReplicaSet ensures that a specified number of pod replicas are running at any given time. It's the mechanism behind Deployments.

Key Functions:
• Maintains a stable set of replica Pods
• Ensures desired number of Pods are always running
• Replaces Pods that fail or are deleted
• Used by Deployments for scaling

🌍 Real-World Example

Think of a ReplicaSet as a factory supervisor:

  • Factory needs exactly 5 workers on the floor
  • If a worker leaves, supervisor hires a replacement immediately
  • If there are 6 workers, supervisor sends one home
  • The supervisor constantly maintains exactly 5 workers

💻 ReplicaSet Example

apiVersion: apps/v1 kind: ReplicaSet metadata: name: nginx-replicaset spec: replicas: 3 # Desired number of pods selector: matchLabels: app: nginx template: metadata: labels: app: nginx spec: containers: - name: nginx image: nginx:latest

⚠️ Common Pitfalls

1. Don't create ReplicaSets directly in production
✅ Use Deployments instead (they manage ReplicaSets for you)
❌ ReplicaSets don't support rolling updates

2. Selector must match pod labels
If labels don't match, ReplicaSet won't manage the pods

3. Deleting a ReplicaSet deletes its Pods
Use --cascade=orphan to keep pods

2.5 Scaling Workloads Foundation
▼

📘 Scaling in Kubernetes

Scaling is the process of increasing or decreasing the number of pod replicas to handle varying load.

Types of Scaling:
• Manual Scaling: You manually set replica count
• Horizontal Pod Autoscaling (HPA): Automatic based on CPU/memory
• Vertical Scaling: Increase pod resources (CPU/RAM)
• Cluster Autoscaling: Add/remove nodes automatically

💻 Manual Scaling

# Scale deployment to 5 replicas kubectl scale deployment webapp --replicas=5 # Scale using YAML file kubectl scale -f deployment.yaml --replicas=3 # Check current replicas kubectl get deployment webapp

🛠️ Try It on AWS/EKS

Scale a deployment and watch it happen:

# Create deployment kubectl create deployment nginx --image=nginx --replicas=2 # Watch pods in real-time kubectl get pods -w # In another terminal, scale up kubectl scale deployment nginx --replicas=5 # Scale down kubectl scale deployment nginx --replicas=1

💰 Cost: More replicas = more resource usage (minimal on existing nodes)

⚠️ Common Pitfalls

1. Scaling to 0 is allowed
--replicas=0 stops all pods but keeps the deployment

2. Manual scaling overrides HPA
If HPA is enabled, manual scaling creates conflicts

3. Nodes must have capacity
Scaling fails if nodes don't have enough CPU/memory

2.6 DaemonSets — One Pod Per Node Important
▼

📘 What is a DaemonSet?

A DaemonSet ensures that all (or some) nodes run a copy of a Pod. As nodes are added to the cluster, Pods are added to them. As nodes are removed, those Pods are garbage collected.

Common Use Cases:
• Logging agents: Fluentd, Logstash on every node
• Monitoring agents: Prometheus node exporter, Datadog agent
• Storage daemons: Ceph, GlusterFS on every node
• Network plugins: kube-proxy, CNI plugins

🌍 Real-World Analogy

Think of a DaemonSet as security cameras in a building:

  • Every floor (node) needs exactly one camera (pod)
  • When a new floor is built, install a camera automatically
  • If a floor is demolished, remove its camera
  • You don't need multiple cameras per floor

💻 DaemonSet Example

apiVersion: apps/v1 kind: DaemonSet metadata: name: fluentd-daemonset spec: selector: matchLabels: name: fluentd template: metadata: labels: name: fluentd spec: containers: - name: fluentd image: fluentd:latest volumeMounts: - name: varlog mountPath: /var/log volumes: - name: varlog hostPath: path: /var/log

🔍 DaemonSet vs Deployment

DaemonSet:
• One pod per node (automatically)
• Used for node-level services
• No replica count needed

Deployment:
• Multiple pods across nodes
• Used for application workloads
• You specify replica count
# Create DaemonSet kubectl apply -f daemonset.yaml # Check DaemonSet kubectl get daemonsets # See which nodes have the pod kubectl get pods -o wide

🛠️ Try It on AWS/EKS

Deploy a monitoring agent on all nodes:

# Create a simple DaemonSet kubectl apply -f - <

💰 Cost: Minimal (one small pod per node)

⚠️ Common Pitfalls

1. DaemonSets ignore pod scheduling preferences by default
They run on ALL nodes unless you use nodeSelector or taints/tolerations

2. DaemonSet pods run even on master nodes
Unless master nodes have taints that prevent scheduling

3. You cannot scale DaemonSets
One pod per node is the rule — scaling doesn't apply

2.7 Pod Scheduling — nodeName & nodeSelector Foundation
▼

📘 Manual Pod Scheduling

By default, the kube-scheduler assigns pods to nodes. But you can manually control where pods run using nodeName or nodeSelector.

1️⃣ Using nodeName (Direct Assignment)

nodeName — Assign pod to a specific node
• Bypasses the scheduler entirely
• Pod scheduled immediately to the named node
• If node doesn't exist or is unavailable, pod fails
• Use with caution!
apiVersion: v1 kind: Pod metadata: name: nginx-manual spec: nodeName: worker-node-1 # Pod MUST run on this node containers: - name: nginx image: nginx

2️⃣ Using nodeSelector (Label-Based)

nodeSelector — Assign pod to nodes with specific labels
• More flexible than nodeName
• Scheduler finds a node matching the labels
• Pod won't schedule if no nodes match
• Recommended approach
# First, label your nodes kubectl label nodes worker-node-1 disktype=ssd kubectl label nodes worker-node-2 disktype=hdd # Create pod with nodeSelector apiVersion: v1 kind: Pod metadata: name: nginx-ssd spec: nodeSelector: disktype: ssd # Pod runs only on nodes with this label containers: - name: nginx image: nginx

🔍 nodeName vs nodeSelector

# Check node labels kubectl get nodes --show-labels # Add label to node kubectl label nodes environment=production # Remove label from node kubectl label nodes environment-

🛠️ Try It on AWS/EKS

Schedule pods to specific node types:

# Get your EKS nodes kubectl get nodes # Label a node (use actual node name) kubectl label nodes workload=frontend # Create pod with nodeSelector kubectl apply -f - <

💰 Cost: No additional cost

⚠️ Common Pitfalls

1. nodeName bypasses scheduler completely
Pod fails if node doesn't exist or has no capacity

2. Pod stays Pending if nodeSelector doesn't match
Check node labels: kubectl get nodes --show-labels

3. nodeSelector is simple but limited
For complex rules, use Node Affinity (not covered in basic test)

2.8 Restart Policies Foundation
▼

📘 What is restartPolicy?

The restartPolicy defines what Kubernetes should do when a container in a pod exits or fails.

Three Restart Policies:
• Always (default) — Always restart, even on success
• OnFailure — Restart only if container exits with error
• Never — Never restart, regardless of exit status

1️⃣ Always (Default)

The container will always restart, even if it exits successfully (exit code 0).

apiVersion: v1 kind: Pod metadata: name: pod-always spec: restartPolicy: Always # This is the default containers: - name: app image: busybox command: ["sh", "-c", "echo 'Hello'; sleep 10"] # Container will restart after sleeping for 10 seconds

2️⃣ OnFailure

The container restarts only if it fails (non-zero exit code). Success = no restart.

apiVersion: v1 kind: Pod metadata: name: pod-onfailure spec: restartPolicy: OnFailure containers: - name: job image: busybox command: ["sh", "-c", "exit 1"] # Container will restart because exit 1 = failure

3️⃣ Never

The container never restarts, regardless of exit status. Used for one-time tasks.

apiVersion: v1 kind: Pod metadata: name: pod-never spec: restartPolicy: Never containers: - name: task image: busybox command: ["sh", "-c", "echo 'Task complete'"] # Container runs once and stops. Never restarts.

🔍 When to Use Which Policy

Always: Long-running applications (web servers, APIs)
OnFailure: Batch jobs that should retry on failure
Never: One-time initialization tasks, data migrations

💻 Testing Restart Policies

# Create pod that fails kubectl run test-restart --image=busybox --restart=OnFailure \ -- sh -c "exit 1" # Watch the restarts (RESTARTS column increases) kubectl get pods test-restart --watch # Check restart count kubectl describe pod test-restart | grep -i restart

🛠️ Try It on AWS/EKS

Test different restart policies:

# Test OnFailure - container will restart kubectl run test-fail --image=busybox --restart=OnFailure \ -- sh -c "echo 'Attempt'; exit 1" # Watch it restart multiple times kubectl get pods test-fail --watch # Test Never - container runs once kubectl run test-never --image=busybox --restart=Never \ -- sh -c "echo 'Done'; exit 0" # Check status (should be Completed) kubectl get pod test-never

💰 Cost: Minimal resource usage

⚠️ Common Pitfalls

1. Always restarts even on success
For one-time jobs, use OnFailure or Never

2. Restart policy applies to ALL containers in pod
You cannot have different policies for different containers

3. CrashLoopBackOff after multiple failures
Kubernetes adds exponential backoff (10s, 20s, 40s... up to 5min)
