1.1 Introduction to Kubernetes Architecture Foundation
▼

📘 What is Kubernetes?

Kubernetes is an open-source container orchestration platform that automates the deployment, scaling, and management of containerized applications. Think of it as an operating system for your cluster.

Source: IBM Technology (6:00)

🏗️ High-Level Architecture

Kubernetes follows a control plane + worker nodes architecture:

Control Plane (Master) — The brain of the cluster
• Makes decisions about the cluster (scheduling, detecting events)
• Components: API Server, etcd, Scheduler, Controller Manager

Worker Nodes — The muscle of the cluster
• Run your actual applications (pods)
• Components: kubelet, kube-proxy, container runtime

1.2 The API Server — The Front Door Essential
▼

📘 What is the API Server?

The Kubernetes API Server (kube-apiserver) is the central hub of the control plane. Every component talks to the API Server — it's the only component that directly communicates with etcd.

Key Functions:
• Exposes the Kubernetes API (REST interface)
• Authenticates and authorizes requests
• Validates and processes API objects
• The only component that writes to etcd

🌍 Real-World Example

When you run kubectl get pods:

  1. kubectl sends an HTTP GET request to the API Server
  2. API Server authenticates you (checks your kubeconfig)
  3. API Server authorizes you (checks RBAC permissions)
  4. API Server queries etcd for pod information
  5. API Server returns the result to kubectl
# Check if API Server is running kubectl get componentstatuses # View API Server logs (if you have access to control plane) kubectl logs -n kube-system kube-apiserver-master-node

⚠️ Common Pitfalls

1. Never bypass the API Server
❌ Don't write directly to etcd — always go through the API Server

2. API Server is a single point of failure
In production, run multiple API Server replicas for high availability

1.3 etcd — The Cluster Database Essential
▼

📘 What is etcd?

etcd is a distributed key-value store that stores all cluster data. It's the single source of truth for the cluster state.

What etcd stores:
• Cluster configuration
• Current state of all resources (pods, services, etc.)
• Secrets and ConfigMaps
• Resource metadata

Key characteristics:
• Consistent and highly-available
• Uses Raft consensus algorithm
• Only the API Server writes to it

🌍 Real-World Example

Think of etcd as the database of your cluster. When you create a deployment with 3 replicas, that desired state (3 replicas) is written to etcd. The controller manager constantly reads from etcd and ensures the actual state matches.

# Check etcd health (requires access to etcd) ETCDCTL_API=3 etcdctl endpoint health # List all keys in etcd (use carefully!) ETCDCTL_API=3 etcdctl get / --prefix --keys-only

⚠️ Common Pitfalls

1. etcd is critical — backup regularly!
Loss of etcd = loss of entire cluster state

2. Never write directly to etcd
Always use the API Server to modify cluster state

3. etcd requires odd number of nodes
For HA: use 3, 5, or 7 etcd nodes (never 2, 4, 6)

🛠️ AWS/EKS Context

In EKS, AWS manages etcd for you. You cannot directly access it, but you can:

# Verify cluster state is persisted kubectl get all --all-namespaces # Check cluster info (includes etcd endpoint if self-managed) kubectl cluster-info dump

💰 Cost: etcd management included in EKS control plane cost ($0.10/hour)

1.4 kube-scheduler — The Pod Matchmaker Important
▼

📘 What is the kube-scheduler?

The kube-scheduler watches for newly created pods that have no assigned node and selects the best node for them to run on.

Scheduling Process:
1. Watch: Detect new pods with no nodeName
2. Filter: Remove unsuitable nodes (insufficient resources, taints, etc.)
3. Score: Rank remaining nodes by suitability
4. Bind: Assign pod to the highest-scoring node

🌍 Real-World Analogy

Imagine you're organizing a conference and need to assign speakers to rooms:

  • Filter: Remove rooms that are too small or already full
  • Score: Prefer rooms with better A/V equipment, closer to attendees
  • Bind: Assign the speaker to the best room

The scheduler does the same for pods and nodes!

🎯 Scheduling Factors

The scheduler considers many factors when choosing a node:

Filtering Criteria:
• Does the node have enough CPU/memory?
• Does the node have required volumes?
• Does the pod's nodeSelector match node labels?
• Are there taints that the pod doesn't tolerate?

Scoring Criteria:
• Resource availability (prefer less-used nodes)
• Pod affinity/anti-affinity rules
• Data locality (prefer nodes with needed data)
• Spread pods across nodes for high availability

💻 Watching the Scheduler

# Watch the scheduler in action kubectl get events --watch # Create a pod and see scheduling events kubectl run nginx --image=nginx # See which node a pod was scheduled to kubectl get pod nginx -o wide # Check pod events (shows scheduling decisions) kubectl describe pod nginx

🚫 Bypassing the Scheduler

You can bypass the scheduler using nodeName in the pod spec:

apiVersion: v1 kind: Pod metadata: name: manual-pod spec: nodeName: worker-node-1 # Bypass scheduler, go directly to this node containers: - name: nginx image: nginx
⚠️ Warning: Using nodeName bypasses all scheduler checks!
• Pod fails if node doesn't exist
• Pod fails if node has no capacity
• Use only for advanced debugging

⚠️ Common Pitfalls

1. Scheduler doesn't move running pods
Once scheduled, pods stay on their node unless deleted or evicted

2. Pod stays Pending if no suitable node exists
Check: kubectl describe pod <name> for scheduling failures

3. You can bypass the scheduler (use carefully!)
Using nodeName in pod spec skips all scheduling logic

1.5 kube-controller-manager — The Reconciliation Loop Important
▼

📘 What is the kube-controller-manager?

The kube-controller-manager runs multiple controllers that watch the cluster state and make changes to move the current state toward the desired state.

Key Controllers Include:
• Node Controller: Monitors node health, marks nodes as unhealthy
• Replication Controller: Ensures correct number of pod replicas
• Endpoints Controller: Populates Endpoints objects (joins Services & Pods)
• Service Account Controller: Creates default ServiceAccounts for namespaces
• Deployment Controller: Manages ReplicaSets and rolling updates

🔄 The Reconciliation Loop

Controllers continuously run a reconciliation loop:

  1. Observe: Read current state from API server
  2. Compare: Compare current state vs desired state (from etcd)
  3. Act: Make changes to reconcile the difference
  4. Repeat: Loop continuously (every few seconds)

🌍 Real-World Analogy

Think of controllers as cruise control in a car:

  • Desired state: "Drive at 65 mph" (you set the cruise control)
  • Current state: Car is going 60 mph (going uphill)
  • Controller action: Increase throttle to reach 65 mph
  • Continuous monitoring: Keep adjusting to maintain 65 mph

💡 Example: ReplicaSet Controller

Let's say you create a Deployment with 3 replicas:

# You create a deployment with 3 replicas kubectl create deployment webapp --image=nginx --replicas=3 # Desired state (stored in etcd): 3 pods # Current state: 0 pods # ReplicaSet Controller sees the difference and: # 1. Creates 3 pods # 2. Monitors them continuously # 3. If a pod dies, creates a replacement immediately

🔍 Seeing Controllers in Action

# Create deployment with 3 replicas kubectl create deployment test --image=nginx --replicas=3 # Watch pods kubectl get pods --watch # In another terminal, delete a pod kubectl delete pod # Controller immediately creates a replacement pod! # This is the ReplicaSet Controller at work

⚠️ Common Pitfalls

1. Controllers react to changes, not commands
They watch state, not kubectl commands

2. Multiple controllers can conflict
Don't manage the same resources with different controllers

3. Controller manager is a single binary
All controllers run in one process (kube-controller-manager)

1.6 kubelet — The Node Agent Essential
▼

📘 What is the kubelet?

The kubelet is an agent that runs on every worker node. It ensures that containers are running in pods as specified.

Key Responsibilities:
• Registers the node with the API server
• Watches for pod assignments to its node
• Pulls container images from registries
• Starts and stops containers
• Reports pod and node status to API server
• Performs health checks (liveness/readiness probes)

🌍 Real-World Analogy

Think of kubelet as a construction site foreman:

  • Receives orders: "Build these houses (pods) on this lot (node)"
  • Manages workers: Tells container runtime to start/stop containers
  • Reports status: "Houses are built and occupied" → API server
  • Health checks: Regularly inspects buildings, reports issues

🔄 kubelet Workflow

  1. Kubelet watches API server for pods scheduled to its node
  2. When a new pod is assigned, kubelet tells container runtime to pull images
  3. Container runtime starts containers according to pod spec
  4. Kubelet monitors container health continuously
  5. Kubelet reports pod status back to API server

💻 Working with kubelet

# Check node status (kubelet reports this) kubectl get nodes # Describe node (shows kubelet info) kubectl describe node # Check kubelet logs (on the node itself) # SSH to the node first journalctl -u kubelet -f # Check kubelet status systemctl status kubelet

🔍 What kubelet Does NOT Do

kubelet does NOT:
• Schedule pods (that's the scheduler)
• Make scheduling decisions
• Manage control plane components
• Handle networking between nodes (that's CNI/kube-proxy)

⚠️ Common Pitfalls

1. kubelet only manages pods assigned to its node
It doesn't know about pods on other nodes

2. If kubelet stops, pods on that node die
Critical component — monitor kubelet health closely

3. kubelet uses container runtime (Docker, containerd)
If container runtime fails, kubelet can't start containers

1.7 kube-proxy — Network Rules Manager Important
▼

📘 What is kube-proxy?

kube-proxy is a network proxy that runs on each node. It maintains network rules that allow network communication to Pods from inside or outside the cluster.

Key Responsibilities:
• Watches the API server for Services and Endpoints
• Maintains network rules (iptables/ipvs) on nodes
• Enables Service load balancing
• Routes traffic from Service IPs to Pod IPs
• Implements ClusterIP, NodePort, and LoadBalancer services

🌍 Real-World Analogy

Think of kube-proxy as a receptionist at a company:

  • Service: "I need to speak to the Sales department"
  • kube-proxy: "Sales has 3 people (pods) available. I'll connect you to Jane"
  • Next call → connects to John (load balancing)
  • If Jane quits (pod dies) → routes calls to John and Sarah only

🔧 How kube-proxy Works

When you create a Service, kube-proxy sets up network rules:

# Example: Create a service kubectl create service clusterip my-service --tcp=80:80 # What kube-proxy does on each node: # 1. Watches for new Service (my-service) # 2. Gets the Service ClusterIP (e.g., 10.96.0.10) # 3. Creates iptables rules: # "Traffic to 10.96.0.10:80 → load balance to pod IPs" # 4. Updates rules when pods are added/removed

⚙️ kube-proxy Modes

Three Operating Modes:

1. iptables mode (default):
• Uses Linux iptables for packet filtering
• Random load balancing
• Most common mode

2. ipvs mode:
• Uses IP Virtual Server (better performance)
• More load balancing algorithms
• Requires kernel support

3. userspace mode (legacy):
• Older, slower mode
• Rarely used today

🔍 Checking kube-proxy

# kube-proxy runs as DaemonSet in kube-system namespace kubectl get pods -n kube-system | grep kube-proxy # Check kube-proxy logs kubectl logs -n kube-system kube-proxy-xxxxx # Check iptables rules created by kube-proxy (on node) # SSH to node first sudo iptables -t nat -L -n | grep

💻 kube-proxy in Action

# Create deployment and service kubectl create deployment nginx --image=nginx --replicas=3 kubectl expose deployment nginx --port=80 # Get service ClusterIP kubectl get svc nginx # kube-proxy automatically: # 1. Created iptables rules for the ClusterIP # 2. Routes traffic to the 3 nginx pods # 3. Load balances across all pods # Test from any pod in cluster: kubectl run test --image=busybox -it --rm -- wget -O- http://nginx

⚠️ Common Pitfalls

1. kube-proxy is NOT a proxy server
Despite the name, it doesn't proxy traffic — it programs network rules

2. kube-proxy runs on ALL nodes
It's a DaemonSet — one instance per node

3. Service networking breaks if kube-proxy fails
Pods can still communicate directly, but Services won't work

1.8 kubectl — Your Command Line Tool Essential
▼

📘 What is kubectl?

kubectl (Kube Control) is the command-line interface for interacting with Kubernetes clusters. It's your primary tool for managing applications.

Source: TechWorld with Nana

🔧 Essential kubectl Commands

1. Get Resources

# List all pods in current namespace kubectl get pods # List all pods in all namespaces kubectl get pods --all-namespaces kubectl get pods -A # Get detailed output kubectl get pods -o wide # Get YAML definition kubectl get pod my-pod -o yaml

2. Describe Resources

# Get detailed information about a pod kubectl describe pod my-pod # Describe includes events - very useful for troubleshooting! kubectl describe node worker-node-1

3. Create Resources

# Imperative - create directly kubectl run nginx --image=nginx # Declarative - create from YAML kubectl apply -f deployment.yaml # Create deployment kubectl create deployment webapp --image=nginx

🛠️ Try It on AWS/EKS

Step 1: Install kubectl

# macOS brew install kubectl # Linux curl -LO "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl" chmod +x kubectl sudo mv kubectl /usr/local/bin/

Step 2: Configure kubectl for EKS

# Install AWS CLI first pip install awscli # Configure kubeconfig for your EKS cluster aws eks update-kubeconfig --name my-cluster --region us-east-1 # Verify connection kubectl get nodes

💰 Cost: kubectl is free

⚠️ Common Pitfalls

1. Wrong context
Always verify: kubectl config current-context

2. Forgetting namespace
Use -n namespace or --all-namespaces

3. Imperative vs Declarative
✅ Use YAML files for production
⚠️ Use commands for quick testing only
