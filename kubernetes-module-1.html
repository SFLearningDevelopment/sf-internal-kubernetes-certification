1.1 Introduction to Kubernetes Architecture Foundation
▼

📘 What is Kubernetes?

Kubernetes is an open-source container orchestration platform that automates the deployment, scaling, and management of containerized applications. Think of it as an operating system for your cluster.

Source: TechWorld with Nana

🏗️ High-Level Architecture

Kubernetes follows a control plane + worker nodes architecture:

Control Plane (Master) — The brain of the cluster
• Makes decisions about the cluster (scheduling, detecting events)
• Components: API Server, etcd, Scheduler, Controller Manager

Worker Nodes — The muscle of the cluster
• Run your actual applications (pods)
• Components: kubelet, kube-proxy, container runtime

🌍 Real-World Analogy

Think of Kubernetes like an orchestra:

  • API Server = The conductor (coordinates everything)
  • etcd = Sheet music (stores the score/state)
  • Scheduler = Stage manager (assigns musicians to positions)
  • Controllers = Section leaders (ensure musicians play correctly)
  • Worker Nodes = Musicians (do the actual work)

⚠️ Common Pitfall

Mistake: Confusing "cluster" with "node"
• A cluster = Control plane + all worker nodes
• A node = A single machine (VM or physical) in the cluster

1.2 The API Server — The Front Door Critical
▼

📘 What is the API Server?

The Kubernetes API Server (kube-apiserver) is the central hub of the control plane. Every component talks to the API Server — it's the only component that directly communicates with etcd.

Key Functions:
• Exposes the Kubernetes API (REST interface)
• Authenticates and authorizes requests
• Validates and processes API objects
• The only component that writes to etcd

🌍 Real-World Example

When you run kubectl get pods:

  1. kubectl sends an HTTP GET request to the API Server
  2. API Server authenticates you (checks your kubeconfig)
  3. API Server authorizes you (checks RBAC permissions)
  4. API Server queries etcd for pod information
  5. API Server returns the result to kubectl
# Check if API Server is running kubectl get componentstatuses # View API Server logs (if you have access to control plane) kubectl logs -n kube-system kube-apiserver-master-node

⚠️ Common Pitfalls

1. Never bypass the API Server
❌ Don't write directly to etcd — always go through the API Server

2. API Server is a single point of failure
In production, run multiple API Server replicas for high availability

1.3 etcd — The Cluster Database Critical
▼

📘 What is etcd?

etcd is a distributed key-value store that stores all cluster data. It's the single source of truth for the cluster state.

What etcd stores:
• Cluster configuration
• Current state of all resources (pods, services, etc.)
• Secrets and ConfigMaps
• Resource metadata

Key characteristics:
• Consistent and highly-available
• Uses Raft consensus algorithm
• Only the API Server writes to it

🌍 Real-World Example

Think of etcd as the database of your cluster. When you create a deployment with 3 replicas, that desired state (3 replicas) is written to etcd. The controller manager constantly reads from etcd and ensures the actual state matches.

# Check etcd health (requires access to etcd) ETCDCTL_API=3 etcdctl endpoint health # List all keys in etcd (use carefully!) ETCDCTL_API=3 etcdctl get / --prefix --keys-only

⚠️ Common Pitfalls

1. etcd is critical — backup regularly!
Loss of etcd = loss of entire cluster state

2. Never write directly to etcd
Always use the API Server to modify cluster state

3. etcd requires odd number of nodes
For HA: use 3, 5, or 7 etcd nodes (never 2, 4, 6)

🛠️ AWS/EKS Context

In EKS, AWS manages etcd for you. You cannot directly access it, but you can:

# Verify cluster state is persisted kubectl get all --all-namespaces # Check cluster info (includes etcd endpoint if self-managed) kubectl cluster-info dump

💰 Cost: etcd management included in EKS control plane cost ($0.10/hour)

1.8 kubectl — Your Command Line Tool Critical
▼

📘 What is kubectl?

kubectl (Kube Control) is the command-line interface for interacting with Kubernetes clusters. It's your primary tool for managing applications.

Source: TechWorld with Nana

🔧 Essential kubectl Commands

1. Get Resources

# List all pods in current namespace kubectl get pods # List all pods in all namespaces kubectl get pods --all-namespaces kubectl get pods -A # Get detailed output kubectl get pods -o wide # Get YAML definition kubectl get pod my-pod -o yaml

2. Describe Resources

# Get detailed information about a pod kubectl describe pod my-pod # Describe includes events - very useful for troubleshooting! kubectl describe node worker-node-1

3. Create Resources

# Imperative - create directly kubectl run nginx --image=nginx # Declarative - create from YAML kubectl apply -f deployment.yaml # Create deployment kubectl create deployment webapp --image=nginx

🛠️ Try It on AWS/EKS

Step 1: Install kubectl

# macOS brew install kubectl # Linux curl -LO "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl" chmod +x kubectl sudo mv kubectl /usr/local/bin/

Step 2: Configure kubectl for EKS

# Install AWS CLI first pip install awscli # Configure kubeconfig for your EKS cluster aws eks update-kubeconfig --name my-cluster --region us-east-1 # Verify connection kubectl get nodes

💰 Cost: kubectl is free

⚠️ Common Pitfalls

1. Wrong context
Always verify: kubectl config current-context

2. Forgetting namespace
Use -n namespace or --all-namespaces

3. Imperative vs Declarative
✅ Use YAML files for production
⚠️ Use commands for quick testing only
