5.1 Troubleshooting Workflow Essential
▼

📘 Systematic Troubleshooting Approach

When things go wrong in Kubernetes, follow a systematic workflow to identify and fix issues quickly.

Source: TechWorld with Nana (14:28)

🔍 The 5-Step Debugging Process

  1. Identify the problem: What's not working?
  2. Check pod status: Is it Running, Pending, or CrashLoopBackOff?
  3. Examine logs: What do the application logs say?
  4. Describe resources: What do events tell you?
  5. Test connectivity: Can pods communicate?
Essential Commands for Every Issue:
• kubectl get pods - Check status
• kubectl logs <pod> - Read logs
• kubectl describe pod <pod> - See events
• kubectl exec -it <pod> -- sh - Interactive debugging

🎯 Quick Diagnostic Chart

Pod Status → What to Check ------------- → --------------- Pending → Events, Node resources, Scheduling issues Running → Logs, Readiness probes CrashLoopBackOff → Logs, Container exit codes ImagePullBackOff → Image name, Registry credentials Error → Logs, Resource limits Terminating → Finalizers, Grace period
5.2 kubectl logs — Reading Application Logs Essential
▼

📘 Understanding kubectl logs

The kubectl logs command shows the stdout/stderr output from containers. It's your first stop for debugging application issues.

What logs show:
• Application stdout/stderr output
• Errors, warnings, info messages
• Stack traces and exceptions
• Custom application logging

💻 Essential Log Commands

# View logs from a pod kubectl logs my-pod # View logs from specific container in multi-container pod kubectl logs my-pod -c container-name # Follow logs in real-time (like tail -f) kubectl logs -f my-pod # Get last 100 lines kubectl logs my-pod --tail=100 # Logs from previous container instance (after crash) kubectl logs my-pod --previous # Logs with timestamps kubectl logs my-pod --timestamps # Logs from last hour kubectl logs my-pod --since=1h

🔍 Common Log Patterns

# Search logs for errors kubectl logs my-pod | grep -i error # Count error occurrences kubectl logs my-pod | grep -c "ERROR" # View logs from all pods with label kubectl logs -l app=webapp --all-containers=true

🛠️ Try It on AWS/EKS

# Create a pod that logs continuously kubectl run logger --image=busybox -- sh -c "while true; do echo $(date) - Hello from logger; sleep 2; done" # Watch logs in real-time kubectl logs -f logger # In another terminal, check last 10 lines kubectl logs logger --tail=10 # Check logs with timestamps kubectl logs logger --timestamps --tail=5

⚠️ Common Pitfalls

1. Logs are lost when containers are deleted
Use centralized logging (CloudWatch, ELK, etc.) for production

2. --previous only works if pod restarted
If pod was deleted and recreated, previous logs are gone

3. Multi-container pods require -c flag
Without it, you'll get an error

5.3 kubectl describe — Detailed Resource Info Essential
▼

📘 Understanding kubectl describe

kubectl describe shows detailed information about Kubernetes resources, including events, which are crucial for troubleshooting.

What describe shows:
• Resource metadata and spec
• Current status and conditions
• Events (most important for debugging!)
• Relationships to other resources

💻 Using kubectl describe

# Describe a pod (shows events at bottom) kubectl describe pod my-pod # Describe a node kubectl describe node worker-node-1 # Describe a service kubectl describe service my-service # Describe deployment kubectl describe deployment my-app

🔍 Reading Events

Events are at the bottom of describe output - they show what Kubernetes tried to do:

Events: Type Reason Age From Message ---- ------ ---- ---- ------- Normal Scheduled 2m default-scheduler Successfully assigned pod Normal Pulling 2m kubelet Pulling image "nginx:latest" Normal Pulled 1m kubelet Successfully pulled image Normal Created 1m kubelet Created container Normal Started 1m kubelet Started container

🚨 Common Error Events

Failed to pull image:
• Image name is wrong
• Image doesn't exist
• No credentials for private registry

Failed scheduling:
• No nodes have enough CPU/memory
• Node selector doesn't match
• Taints prevent scheduling

Container failed:
• Application crashed
• Health checks failing
• Missing environment variables

🛠️ Try It on AWS/EKS

# Create a pod with a bad image kubectl run broken --image=nginx:nonexistent-tag # Describe it to see the error kubectl describe pod broken # Look for events showing ImagePullBackOff # Events section will show: # "Failed to pull image ... not found"

5.4 kubectl exec — Interactive Debugging Important
▼

📘 Understanding kubectl exec

kubectl exec executes commands inside a running container. Use it to debug from inside the container.

Common uses:
• Run diagnostic commands inside container
• Test network connectivity
• Check file systems and permissions
• Inspect running processes
• Verify environment variables

💻 Using kubectl exec

# Run a single command kubectl exec my-pod -- ls /app # Interactive shell (most common) kubectl exec -it my-pod -- /bin/bash kubectl exec -it my-pod -- sh # If bash not available # For multi-container pods kubectl exec -it my-pod -c container-name -- /bin/bash # Run command and see output kubectl exec my-pod -- env kubectl exec my-pod -- ps aux kubectl exec my-pod -- cat /etc/resolv.conf

🔧 Common Debugging Commands

# Inside the pod shell: # Check network connectivity ping google.com curl http://service-name nslookup service-name # Check processes ps aux top # Check disk space df -h # Check environment variables env | grep DATABASE # Test application endpoint curl http://localhost:8080/health # Check listening ports netstat -tulpn

🛠️ Try It on AWS/EKS

# Create a pod kubectl run debug-pod --image=nginx # Get a shell inside the pod kubectl exec -it debug-pod -- /bin/bash # Inside the pod, test things: # Check what's running ps aux # Test DNS nslookup kubernetes.default # Check environment env # Exit exit

⚠️ Common Pitfalls

1. Container must be running
Can't exec into CrashLoopBackOff containers

2. Not all images have /bin/bash
Try /bin/sh or install tools using package managers

3. Commands run as container's user
May not have root permissions

5.5 kubectl get events — Cluster-Wide Events Foundation
▼

📘 Cluster Events

Events are time-ordered records of what happened in the cluster. They're invaluable for understanding the sequence of actions.

# View all events in current namespace kubectl get events # Sort by timestamp kubectl get events --sort-by='.lastTimestamp' # Watch events in real-time kubectl get events --watch # Events for specific resource kubectl get events --field-selector involvedObject.name=my-pod # All events across all namespaces kubectl get events --all-namespaces

⚠️ Event Retention

Important: Events are automatically deleted after 1 hour by default!
For long-term tracking, use a logging solution.
5.6 CrashLoopBackOff — Container Restart Issues Essential
▼

📘 What is CrashLoopBackOff?

CrashLoopBackOff means the container keeps crashing and Kubernetes keeps trying to restart it with exponential backoff.

Common Causes:
• Application crashes immediately on startup
• Missing environment variables or config files
• Wrong command or entrypoint
• Port already in use
• Application needs dependencies that aren't ready

🔍 Debugging CrashLoopBackOff

# Step 1: Check pod status kubectl get pod my-pod # STATUS: CrashLoopBackOff # Step 2: Check logs from crashed container kubectl logs my-pod --previous # Step 3: Describe pod for events kubectl describe pod my-pod # Step 4: Check if it's a config issue kubectl get pod my-pod -o yaml | grep -A 10 env

🛠️ Common Fixes

Fix 1: Missing environment variables
Add required env vars to pod spec

Fix 2: Wrong command
Verify command and args in container spec

Fix 3: Application not ready
Add init containers or readiness probes

5.7 ImagePullBackOff — Image Issues Essential
▼

📘 What is ImagePullBackOff?

ImagePullBackOff means Kubernetes cannot pull the container image from the registry.

Common Causes:
• Image name is incorrect (typo)
• Image doesn't exist in registry
• Private registry requires credentials
• Network issues reaching registry
• Image tag doesn't exist

🔍 Debugging ImagePullBackOff

# Check events for error message kubectl describe pod my-pod # Look for: # "Failed to pull image... manifest unknown" # "Failed to pull image... authentication required" # Verify image name kubectl get pod my-pod -o yaml | grep image: # Test image pull manually docker pull

🛠️ Solutions

Wrong image name: Fix the image name in deployment
Private registry: Add imagePullSecrets
Missing tag: Specify correct tag or use 'latest'

5.8 Pending Pods — Scheduling Issues Important
▼

📘 Why Pods Stay Pending

A pod in Pending state means the scheduler cannot find a suitable node to run it.

Common Reasons:
• Insufficient CPU or memory on nodes
• Node selector doesn't match any node
• Taints prevent scheduling
• PVC cannot be bound
• No nodes available

🔍 Debugging Pending Pods

# Check events kubectl describe pod pending-pod # Common event messages: # "0/3 nodes are available: insufficient cpu" # "0/3 nodes match node selector" # "pod has unbound PersistentVolumeClaims" # Check node resources kubectl top nodes kubectl describe nodes

5.9 Node Troubleshooting Important
▼

📘 Checking Node Health

Unhealthy nodes can cause widespread pod failures.

# Check node status kubectl get nodes # STATUS should be "Ready" # If "NotReady" or "Unknown", there's a problem # Describe node for details kubectl describe node # Check node conditions kubectl get nodes -o custom-columns=NAME:.metadata.name,STATUS:.status.conditions[?(@.type=="Ready")].status

5.10 Troubleshooting Checklist Reference
▼

📋 Complete Troubleshooting Checklist

✅ Step-by-Step Debugging Process:

1. Check Pod Status
□ kubectl get pods
□ Is it Running, Pending, CrashLoopBackOff, or ImagePullBackOff?

2. Check Logs
□ kubectl logs <pod>
□ kubectl logs <pod> --previous (if crashed)

3. Check Events
□ kubectl describe pod <pod>
□ Read events at the bottom

4. Check Resources
□ kubectl describe node
□ Are there enough CPU/memory?

5. Check Networking
□ kubectl get endpoints
□ Can pods communicate?

6. Check Configuration
□ Environment variables set correctly?
□ ConfigMaps/Secrets mounted?
□ Volume mounts working?
