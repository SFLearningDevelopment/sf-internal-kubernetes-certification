4.1 Volumes — Pod Storage Foundation
▼

📘 What are Volumes?

Volumes provide storage to containers in a Pod. Unlike container filesystems (which are ephemeral), volumes persist beyond container restarts.

Why Volumes?
• Container filesystems are ephemeral (lost on restart)
• Volumes persist data across container restarts
• Multiple containers in a pod can share volumes
• Many volume types available (emptyDir, hostPath, cloud storage, etc.)

📁 Common Volume Types

1. emptyDir (Temporary Storage)

Created when Pod is assigned to a node, deleted when Pod is removed. Perfect for scratch space.

apiVersion: v1 kind: Pod metadata: name: test-pod spec: containers: - name: app image: nginx volumeMounts: - mountPath: /cache name: cache-volume volumes: - name: cache-volume emptyDir: {} # Temporary storage

2. hostPath (Node Storage)

Mounts a file or directory from the host node's filesystem. Use carefully!

apiVersion: v1 kind: Pod metadata: name: hostpath-pod spec: containers: - name: app image: nginx volumeMounts: - mountPath: /data name: host-volume volumes: - name: host-volume hostPath: path: /var/log # Node directory type: Directory
⚠️ hostPath Warning:
• Ties pod to specific node
• Security risk (access to host filesystem)
• Data lost if pod moves to another node
• Use only for special cases (logging, monitoring)

🌍 Real-World Analogy

Think of volumes like storage lockers:

  • emptyDir: Temporary locker (cleared when you leave)
  • hostPath: Shared office drawer (on that desk only)
  • PersistentVolume: Personal storage unit (follows you anywhere)

🛠️ Try It on AWS/EKS

Create a pod with emptyDir volume:

kubectl apply -f - < /cache/data.txt; sleep 3600"] volumeMounts: - name: cache mountPath: /cache - name: reader image: busybox command: ["sh", "-c", "sleep 10; cat /cache/data.txt; sleep 3600"] volumeMounts: - name: cache mountPath: /cache volumes: - name: cache emptyDir: {} EOF # Check logs - reader should see writer's file kubectl logs cache-pod -c reader

💰 Cost: emptyDir uses node disk (no additional cost)

⚠️ Common Pitfalls

1. emptyDir is NOT persistent
Data lost when pod is deleted (but survives container restarts)

2. hostPath ties pod to a node
Pod can't move to another node and access the same data

3. Volumes are defined at pod level, not container
Multiple containers in a pod can share the same volume

4.2 PersistentVolumes (PV) — Cluster Storage Critical
▼

📘 What is a PersistentVolume?

A PersistentVolume (PV) is a piece of storage in the cluster that has been provisioned by an administrator or dynamically provisioned using Storage Classes. It's a cluster resource, just like a node.

Key Characteristics:
• PV exists independently of pods
• Lifecycle independent of any pod using it
• Can be provisioned statically (admin) or dynamically (StorageClass)
• Has a reclaim policy (what happens when released)
• Different access modes and capacity

🌍 Real-World Analogy

Think of PersistentVolumes like rental storage units:

  • PV: The actual storage unit (exists whether rented or not)
  • PVC: Your rental agreement (claims the storage)
  • Pod: You (the person using the storage)
  • When you leave (pod deleted), the storage unit still exists

💻 Creating a PersistentVolume

apiVersion: v1 kind: PersistentVolume metadata: name: my-pv spec: capacity: storage: 10Gi accessModes: - ReadWriteOnce # Can be mounted by one node persistentVolumeReclaimPolicy: Retain hostPath: path: /mnt/data # Example using hostPath (for testing only)

📊 Access Modes

Three Access Modes:

ReadWriteOnce (RWO):
• Volume can be mounted as read-write by a single node
• Most common mode
• Example: AWS EBS, Azure Disk

ReadOnlyMany (ROX):
• Volume can be mounted as read-only by many nodes
• Good for shared configuration

ReadWriteMany (RWX):
• Volume can be mounted as read-write by many nodes
• Example: AWS EFS, Azure Files, NFS
• Required for stateful apps that scale horizontally

♻️ Reclaim Policies

What happens when PVC is deleted:

Retain: PV kept, data preserved, manual cleanup needed
Delete: PV and underlying storage deleted automatically
Recycle (deprecated): Data deleted, PV reused

🔍 Checking PersistentVolumes

# List all PVs in cluster kubectl get pv # Example output: # NAME CAPACITY ACCESS MODES RECLAIM POLICY STATUS # my-pv 10Gi RWO Retain Available # Get PV details kubectl describe pv my-pv # Delete PV kubectl delete pv my-pv

📈 PV Status

PersistentVolumes go through different phases:

  • Available: PV is free, not yet bound to a claim
  • Bound: PV is bound to a PVC
  • Released: PVC deleted, but PV not yet reclaimed
  • Failed: PV has failed automatic reclamation

🛠️ AWS/EKS - Using EBS as PersistentVolume

On EKS, you typically use dynamic provisioning, but here's static:

# 1. Create an EBS volume first (via AWS Console or CLI) aws ec2 create-volume \ --size 10 \ --region us-east-1 \ --availability-zone us-east-1a \ --volume-type gp3 # Note the volume ID (e.g., vol-0abc123...) # 2. Create PV pointing to that EBS volume kubectl apply -f - <

💰 Cost: AWS EBS gp3 ~$0.08/GB/month

⚠️ Common Pitfalls

1. PV is cluster-scoped, not namespaced
PVs are visible across all namespaces

2. Access mode must match storage type
AWS EBS only supports ReadWriteOnce (single node)
AWS EFS supports ReadWriteMany (multi-node)

3. Reclaim policy matters!
"Delete" policy removes data permanently when PVC is deleted

4.3 PersistentVolumeClaims (PVC) — Requesting Storage Critical
▼

📘 What is a PersistentVolumeClaim?

A PersistentVolumeClaim (PVC) is a request for storage by a user. It's like a pod: pods consume node resources, PVCs consume PV resources.

Key Characteristics:
• PVC is namespaced (unlike PV which is cluster-scoped)
• Pods reference PVCs to use storage
• PVC specifies size and access mode needed
• Kubernetes binds PVC to a matching PV automatically
• PVC can trigger dynamic provisioning if no PV matches

🌍 Real-World Analogy

Continuing the storage unit analogy:

  • PV: Storage unit available for rent (100 sq ft)
  • PVC: Your rental request ("I need 50 sq ft of storage")
  • Binding: Storage company assigns you a unit that matches
  • Pod: You move your stuff into the storage unit

💻 Creating a PersistentVolumeClaim

apiVersion: v1 kind: PersistentVolumeClaim metadata: name: my-pvc spec: accessModes: - ReadWriteOnce # Must match available PV resources: requests: storage: 5Gi # Requesting 5GB

🔗 PVC to PV Binding

Kubernetes automatically binds PVCs to PVs based on:

  1. Access Mode: PVC and PV access modes must match
  2. Storage Size: PV must be >= PVC requested size
  3. Storage Class: If specified, must match
  4. Selector: Optional label selectors for specific PVs

🔍 Using PVC in a Pod

apiVersion: v1 kind: Pod metadata: name: pod-using-pvc spec: containers: - name: app image: nginx volumeMounts: - mountPath: /data name: my-storage volumes: - name: my-storage persistentVolumeClaim: claimName: my-pvc # Reference the PVC

📊 Checking PVC Status

# List PVCs in current namespace kubectl get pvc # Example output: # NAME STATUS VOLUME CAPACITY ACCESS MODES # my-pvc Bound my-pv 10Gi RWO # Get detailed PVC information kubectl describe pvc my-pvc # Delete PVC (also releases the PV based on reclaim policy) kubectl delete pvc my-pvc

🎯 PVC Status

PVC States:
• Pending: No suitable PV found yet
• Bound: Successfully bound to a PV
• Lost: PV exists but is not available

🛠️ Try It on AWS/EKS

Complete workflow: PV → PVC → Pod

# Step 1: Create PVC (dynamic provisioning on EKS) kubectl apply -f - < /usr/share/nginx/html/index.html" # Step 7: Delete pod (data persists!) kubectl delete pod app-with-storage # Step 8: Create new pod with same PVC kubectl run app2 --image=nginx --overrides=' { "spec": { "containers": [{ "name": "app2", "image": "nginx", "volumeMounts": [{ "mountPath": "/usr/share/nginx/html", "name": "storage" }] }], "volumes": [{ "name": "storage", "persistentVolumeClaim": { "claimName": "ebs-claim" } }] } }' # Step 9: Data is still there! kubectl exec app2 -- cat /usr/share/nginx/html/index.html

💰 Cost: EBS volume continues charging even when pod is deleted (until PVC is deleted)

⚠️ Common Pitfalls

1. PVC stays Pending if no matching PV
Check: kubectl describe pvc <name> for events
Reasons: No PV with enough capacity, wrong access mode, no StorageClass

2. PVC is namespaced, PV is not
PVC in namespace "app" can only be used by pods in "app" namespace

3. Deleting PVC may delete underlying storage
Check PV's reclaimPolicy before deleting PVC!
• Retain = data safe
• Delete = data gone forever

4.4 Storage Classes & Dynamic Provisioning Important
▼

📘 What is a StorageClass?

A StorageClass provides a way to describe different "classes" of storage. It enables dynamic provisioning - automatically creating PVs when PVCs are created.

Why StorageClass?
• Eliminates need for admins to pre-create PVs manually
• Automatically provisions storage when PVC is created
• Different classes for different performance/cost tiers
• Example: "fast-ssd", "slow-hdd", "standard"
• Cloud providers offer default StorageClasses

🌍 Real-World Analogy

Think of StorageClass like car rental tiers:

  • StorageClass "economy": Basic storage (slow, cheap)
  • StorageClass "premium": Fast SSD storage (expensive)
  • PVC: "I need a premium car" (request)
  • Dynamic Provisioning: Rental company automatically provides a premium car

💻 StorageClass Example

apiVersion: storage.k8s.io/v1 kind: StorageClass metadata: name: fast-ssd provisioner: kubernetes.io/aws-ebs # AWS EBS provisioner parameters: type: gp3 # EBS volume type iopsPerGB: "10" encrypted: "true" volumeBindingMode: WaitForFirstConsumer

🔄 Static vs Dynamic Provisioning

Static Provisioning (Manual)

1. Admin creates PV manually
2. User creates PVC
3. Kubernetes binds PVC to existing PV
4. Pod uses PVC

Problem: Admin must pre-create all PVs

Dynamic Provisioning (Automatic)

1. User creates PVC referencing a StorageClass
2. StorageClass automatically provisions a new PV
3. PVC is bound to the new PV
4. Pod uses PVC

Benefit: No manual admin work!

🎯 Using Dynamic Provisioning

# Create PVC with StorageClass apiVersion: v1 kind: PersistentVolumeClaim metadata: name: dynamic-pvc spec: storageClassName: fast-ssd # Reference StorageClass accessModes: - ReadWriteOnce resources: requests: storage: 10Gi # Kubernetes automatically: # 1. Finds StorageClass "fast-ssd" # 2. Uses the provisioner to create a new PV # 3. Binds the PVC to the new PV

📊 Check StorageClasses

# List all StorageClasses kubectl get storageclass kubectl get sc # Short form # Example output on EKS: # NAME PROVISIONER RECLAIMPOLICY # gp2 (default) kubernetes.io/aws-ebs Delete # Describe StorageClass kubectl describe sc gp2 # Set default StorageClass kubectl patch storageclass gp2 -p '{"metadata": {"annotations":{"storageclass.kubernetes.io/is-default-class":"true"}}}'

🔍 Default StorageClass

If a PVC doesn't specify a StorageClass, Kubernetes uses the default StorageClass (if one exists).

# PVC without storageClassName uses default apiVersion: v1 kind: PersistentVolumeClaim metadata: name: auto-pvc spec: accessModes: - ReadWriteOnce resources: requests: storage: 5Gi # Uses default StorageClass automatically

🛠️ AWS/EKS - Dynamic Provisioning with EBS

EKS comes with a default gp2 StorageClass. Let's use it:

# Step 1: Check available StorageClasses kubectl get sc # Step 2: Create PVC (no PV needed!) kubectl apply -f - <

💰 Cost: EBS gp2 ~$0.10/GB/month (8GB = ~$0.80/month)

⚡ Volume Binding Modes

Immediate (default):
• PV created immediately when PVC is created
• May not be in the same zone as the pod

WaitForFirstConsumer (recommended):
• PV created only when a pod uses the PVC
• Ensures PV is in the same zone as the pod
• Prevents cross-zone mounting errors

⚠️ Common Pitfalls

1. StorageClass provisioner must exist
EKS needs EBS CSI driver for gp3 volumes (gp2 works by default)

2. Dynamic PVs inherit StorageClass reclaim policy
Default is usually "Delete" - PV deleted when PVC is deleted!

3. Different clouds have different provisioners
• AWS: kubernetes.io/aws-ebs (EBS), efs.csi.aws.com (EFS)
• GCP: kubernetes.io/gce-pd
• Azure: kubernetes.io/azure-disk

4. PVC without StorageClass needs manual PV
If no default StorageClass, PVC stays Pending forever

📋 Static vs Dynamic Comparison

When to use Static Provisioning:
• Pre-existing storage you want to use
• Special storage requirements
• Testing/development with hostPath

When to use Dynamic Provisioning:
• Production workloads (recommended)
• Cloud environments (AWS, GCP, Azure)
• When you want automation
• Most common approach today
