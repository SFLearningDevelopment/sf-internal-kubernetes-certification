3.1 Services Overview Foundation
▼

📘 What is a Service?

A Kubernetes Service is an abstraction that defines a logical set of Pods and a policy to access them. Services enable loose coupling between dependent Pods.

Why Services are needed:
• Pods are ephemeral — they can die and be recreated with new IPs
• Services provide a stable endpoint (IP/DNS) to access Pods
• Services enable load balancing across multiple Pods
• Services can expose applications outside the cluster

🌍 Real-World Analogy

Think of a Service like a company phone number:

  • Employees (Pods) come and go, but the phone number stays the same
  • Calls are routed to available employees automatically
  • Customers don't need to know which specific employee will answer
  • The phone system (Service) handles the routing

3.2 Service Types — ClusterIP, NodePort, LoadBalancer Critical
▼

📘 Kubernetes Service Types

Kubernetes offers different Service types for different use cases:

Source: TechWorld with Nana (18:12)

1️⃣ ClusterIP (Default)

ClusterIP — Internal access only
• Exposes service on an internal IP in the cluster
• Only reachable from within the cluster
• Default type if not specified
• Use for: Internal microservices, databases
apiVersion: v1 kind: Service metadata: name: my-service spec: type: ClusterIP # This is the default selector: app: my-app ports: - port: 80 targetPort: 8080

2️⃣ NodePort

NodePort — External access via node IP
• Exposes service on each Node's IP at a static port (30000-32767)
• Accessible from outside the cluster using <NodeIP>:<NodePort>
• Use for: Development, testing, simple external access
apiVersion: v1 kind: Service metadata: name: my-nodeport-service spec: type: NodePort selector: app: my-app ports: - port: 80 targetPort: 8080 nodePort: 30007 # Optional, auto-assigned if omitted

3️⃣ LoadBalancer

LoadBalancer — Cloud load balancer
• Creates an external load balancer (AWS ELB, GCP LB, Azure LB)
• Automatically provisions cloud infrastructure
• Use for: Production external-facing applications
apiVersion: v1 kind: Service metadata: name: my-loadbalancer-service spec: type: LoadBalancer selector: app: my-app ports: - port: 80 targetPort: 8080

🛠️ Try It on AWS/EKS

Create a LoadBalancer Service on EKS:

# Create a deployment first kubectl create deployment webapp --image=nginx # Expose it as LoadBalancer kubectl expose deployment webapp --type=LoadBalancer --port=80 # Check the service (AWS will provision an ELB) kubectl get svc webapp # Get the external URL (takes 2-3 minutes) kubectl get svc webapp -o jsonpath='{.status.loadBalancer.ingress[0].hostname}'

💰 Cost: AWS ELB ~$0.025/hour + data transfer costs

⚠️ Common Pitfalls

1. NodePort valid range: 30000-32767
Trying to use ports outside this range will fail

2. LoadBalancer only works on cloud providers
On local clusters (minikube), LoadBalancer will remain "Pending"

3. ClusterIP is internal only
You cannot access ClusterIP services from outside the cluster

3.3 Service Discovery & DNS Important
▼

📘 Kubernetes DNS

Kubernetes automatically creates DNS records for Services, allowing pods to discover and communicate with services by name instead of IP addresses.

Key Benefits:
• Services get automatic DNS names
• Pods can use service names instead of IPs
• DNS updates automatically when services change
• Works across namespaces

🌐 DNS Naming Format (FQDN)

Every Kubernetes Service gets a Fully Qualified Domain Name (FQDN) in this format:

..svc.cluster.local Examples: webapp.default.svc.cluster.local database.production.svc.cluster.local api.backend.svc.cluster.local
FQDN Breakdown:
• service-name — Name of the Service
• namespace — Kubernetes namespace
• svc — Indicates it's a Service
• cluster.local — Default cluster domain

🔍 Short Names vs Full Names

Pods can use shortened DNS names depending on the namespace:

# Same namespace - use short name curl http://webapp # Same namespace - use namespace curl http://webapp.default # Different namespace - use full FQDN curl http://webapp.production.svc.cluster.local

💻 Testing DNS Resolution

# Create a test service kubectl create service clusterip test-svc --tcp=80:80 # Run a debug pod to test DNS kubectl run debug --image=busybox -it --rm -- sh # Inside the pod, test DNS: nslookup test-svc nslookup test-svc.default nslookup test-svc.default.svc.cluster.local # Test connectivity wget -O- http://test-svc

🛠️ Try It on AWS/EKS

Test service discovery with DNS:

# Create deployment and service kubectl create deployment nginx --image=nginx kubectl expose deployment nginx --port=80 # Create test pod in same namespace kubectl run test --image=busybox -it --rm -- sh # Inside the test pod: wget -O- http://nginx # Short name works! nslookup nginx # See DNS resolution # Exit the pod exit

💰 Cost: DNS is built-in, no additional cost

⚠️ Common Pitfalls

1. DNS only works for Services, not Pods
Individual pod IPs don't get DNS names (use Services)

2. Cross-namespace requires full name
webapp won't work across namespaces — use webapp.other-ns

3. DNS propagation isn't instant
New services may take a few seconds to be resolvable

3.4 Service Selectors & Labels Critical
▼

📘 How Services Find Pods

Services use label selectors to identify which Pods should receive traffic. This is the core mechanism of Kubernetes networking.

Label Selector Workflow:
1. Pods have labels (key-value pairs)
2. Services define selectors (which labels to match)
3. Any Pod with matching labels joins the Service
4. Service load-balances traffic across matching Pods

🏷️ Labels on Pods

apiVersion: v1 kind: Pod metadata: name: webapp-pod labels: app: webapp # Label: app=webapp tier: frontend # Label: tier=frontend version: v1 # Label: version=v1 spec: containers: - name: nginx image: nginx

🎯 Selectors in Services

apiVersion: v1 kind: Service metadata: name: webapp-service spec: selector: app: webapp # Matches pods with label app=webapp ports: - port: 80 targetPort: 80

This Service will send traffic to any Pod that has the label app: webapp, regardless of other labels.

💻 Working with Labels

# View pod labels kubectl get pods --show-labels # Filter pods by label kubectl get pods -l app=webapp # Add label to existing pod kubectl label pod my-pod environment=production # Remove label from pod kubectl label pod my-pod environment- # View service endpoints (shows which pods are selected) kubectl get endpoints webapp-service

🔄 Dynamic Pod Selection

The beauty of selectors: they're dynamic!

  • Add a pod with matching labels → automatically joins the service
  • Remove the label → pod removed from service immediately
  • No need to update the service configuration

🛠️ Try It on AWS/EKS

See label selectors in action:

# Create deployment with labels kubectl create deployment webapp --image=nginx kubectl label deployment webapp app=frontend # Create service with selector kubectl expose deployment webapp --port=80 --selector=app=frontend # Check which pods are selected kubectl get endpoints webapp # Scale deployment - new pods automatically join service kubectl scale deployment webapp --replicas=3 # Check endpoints again - 3 pods now! kubectl get endpoints webapp

💰 Cost: No additional cost

⚠️ Common Pitfalls

1. Labels must match exactly
app: webapp ≠ app: Webapp (case-sensitive)

2. Service with no matching pods = no endpoints
Check: kubectl get endpoints <service-name>

3. Multiple selectors = AND logic
Pods must match ALL selectors, not just one

3.5 NetworkPolicy — Traffic Control Important
▼

📘 What is NetworkPolicy?

NetworkPolicy is a Kubernetes resource that defines rules for controlling network traffic between Pods, similar to a firewall.

Key Concepts:
• Controls ingress (incoming) and egress (outgoing) traffic
• Applied to Pods using label selectors
• Default behavior: all traffic allowed (if no policies exist)
• Requires a CNI plugin that supports NetworkPolicy (Calico, Cilium, etc.)

🌍 Real-World Analogy

Think of NetworkPolicy as building security:

  • By default, anyone can enter any room (no NetworkPolicy)
  • You add badge readers (NetworkPolicy)
  • Only employees with matching badges (labels) can enter
  • You control which rooms they can access (ingress/egress rules)

💻 Basic NetworkPolicy Example

apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: allow-frontend spec: podSelector: matchLabels: app: database # Apply to pods with label app=database policyTypes: - Ingress ingress: - from: - podSelector: matchLabels: app: frontend # Only allow traffic from frontend pods ports: - protocol: TCP port: 5432

This policy allows only pods with label app: frontend to connect to database pods on port 5432.

🚫 Deny All Traffic (Default Deny)

apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: default-deny-all spec: podSelector: {} # Empty selector = applies to all pods policyTypes: - Ingress - Egress # No ingress/egress rules = deny all traffic
⚠️ Important: NetworkPolicy requires a compatible CNI plugin!
• AWS EKS default (VPC CNI) does NOT support NetworkPolicy
• You need to install Calico or Cilium on EKS
• Minikube supports NetworkPolicy by default

⚠️ Common Pitfalls

1. NetworkPolicy is not global
Policies only affect pods in the same namespace

2. Policies are additive
Multiple policies = union of all rules (not intersection)

3. No policy = all traffic allowed
You must explicitly create deny rules
